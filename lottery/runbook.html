Operating Runbook

How the lottery works, where everything lives, and what to do each month.

The stack at a glance

What happens when someone enters

1
Page loads
The page asks the Worker's /status for the live pot, count and promoter details (Worker → Apps Script → Sheet).
2
They pay
Form posts to the Worker's /checkout; the Worker builds a £2×qty Stripe session and redirects them to Stripe. Card details go only to Stripe.
3
Stripe confirms
On success they land on thanks.html. Separately, Stripe sends a checkout.session.completed webhook to the Worker's /webhook — this is the real signal, not the redirect.
4
Worker verifies & forwards
Checks Stripe's signature, then forwards to Apps Script with the shared secret.
5
Apps Script re-checks & writes
Calls back to Stripe to confirm it's genuinely paid, captures the fee, then writes one row per ticket to Entries (under a lock, deduped) and emails the buyer their ticket number(s) and reference.
Why three checks? The browser is never trusted. A paid entry is only written after the Stripe signature, the shared secret, and a fresh re-fetch from Stripe all pass — so a faked or replayed message can't invent a payment.

What happens at the draw

6
Trigger fires
1st of the month, 20:00 (or run runDraw by hand). Picks one ticket row at random — equal odds per ticket.
7
Maths
Gross = tickets × £2 · Prize = 50% · Expenses = fixed costs + Stripe fees · Charity = the rest. Halts if charity would drop below 20%.
8
Records & notifies
Writes the Draws row, creates a one-time 30-day payout token in Payouts, emails the winner a secure link and you the full breakdown, then opens next month.
9
Winner claims, you pay
Winner enters bank details on the secure form → saved to the protected Payouts tab. You pay by bank transfer (manual, two organisers), then mark the row paid.

The reset button

One function clears everything for a fresh start: in the Apps Script editor, pick resetTestData and press Run. It:

  • clears all data rows from Entries / Draws / Events / Payouts (keeps the headers);
  • resets the ticket counters, so numbering restarts at 1;
  • re-opens the current draw row so the engine has somewhere to write.

Config and secrets are left untouched. Use it before the club test, and again as the pre-launch wipe.

Built-in safety. resetTestData refuses to run once a real registration_no is set in Config — so it can never wipe the live, legally-required draw records. Always do the pre-launch reset before you enter the ERYC number. Genuine override (rarely needed): add Script Property FORCE_RESET = yes, then run once.

The club dry run

  1. Reset first with resetTestData so numbers start clean at 1.
  2. Brief the group it's a test: use card 4242 4242 4242 4242, any future expiry/CVC. Warn them they'll get a real confirmation email, and the winner a real "you've won" email.
  3. Share the link and let them enter freely. Watch the pot tick up on the page.
  4. Run the draw — you don't need to wait for the 1st; run runDraw by hand. Check the winner + promoter emails fire and the payout link works.
  5. Test the payout: the "winner" enters dummy bank details (e.g. Test / 00-00-00 / 12345678), you see them land in Payouts, then mark the row paid.
  6. Reset again before going live, so no test data carries over.
Stay in one Stripe environment for the whole test (don't switch sandbox ↔ test mid-run, or webhook deliveries stop — see troubleshooting). Everything lands in the live Sheet, which is why the reset at the end matters.

Two cross-checks catch 90% of "it's not working": GAS_SECRET = SHARED_SECRET (or every webhook is "denied"), and the same Stripe key everywhere — all test, or all live, never mixed.

The monthly routine

  1. Reconcile before the draw. Check the number of paid payments in Stripe for the month matches the number of rows in Entries. If one's short, a webhook blipped — add the missing entry by hand before drawing.
  2. Let the draw run (1st, 20:00) or run runDraw manually. You'll get the winner email with the full breakdown.
  3. Pay the winner. When their bank details arrive (you're emailed), confirm with a second organiser, transfer the prize, then mark the Payouts row paid.
  4. Check the figures. Stripe fees are captured automatically into Draws. Add any other running cost to Config expenses_pence before filing if needed.
  5. File the ERYC Lottery Return within 3 months of the draw — proceeds, prizes, expenses, amount to good causes.
  6. Move the good-causes share to Beverley Round Table funds.
You never change the month by hand. When the draw runs, it automatically closes the current month, opens the next, and updates current_draw_id in Config. No 1st-of-the-month scramble.

Going live (test → real money)

  • ERYC registration approved (have the number ready)
  • Run resetTestData FIRST to clear all test data — do this before entering the registration number (the reset locks itself once a registration number is set)
  • Promoter name + correct contact address in Config (a named BRT member — not the national HQ)
  • registration_no filled in Config (after the reset)
  • Stripe account activated; lottery/charity raffle declared and ERYC registration uploaded
  • Worker secrets swapped to live (sk_live_…, and a whsec_… from a live webhook endpoint)
  • Apps Script STRIPE_SECRET_KEY swapped to the live key
  • current_draw_id set to the launch month
  • Payouts tab protected to the promoter only
  • Run installTriggers once so the monthly draw fires automatically
  • One final transaction reconciles end to end
The golden rule: the sk_live_ key, the live webhook, and the real payments must all be in live mode together. Mixing live and test (or a sandbox) is what makes a payment "succeed but never call back".

If something's not working

SymptomLikely cause & fix
Page loads but pot & footer stay blank (—)/status blocked. Check CORS: ALLOWED_ORIGIN must equal the page's exact origin (no trailing slash), and the page must be served from that domain. Confirm WORKER_URL is set.
Payment succeeds but "No event deliveries" on the webhookEnvironment mismatch — the #1 gotcha. The Stripe key, the payment, and the webhook endpoint must all be in the same environment (all normal Test mode, or all live, or all the same sandbox). A webhook created in a different sandbox/mode — or created after the payment — never receives the event. Put all three together and re-test, or use the event's Resend.
Paid in Stripe but no row in EntriesWebhook problem. Confirm the Stripe webhook points at …/webhook and listens for checkout.session.completed; that test/live keys aren't mixed; and that GAS_SECRET = SHARED_SECRET.
Pot shows 0 / £0.00 but the sheet is full of entriesGoogle Sheets auto-converted the draw_id cells from text to dates. The current code normalises this so it counts correctly anyway; if you ever hand-type a draw id, format the column as plain text first. Also check Config current_draw_id matches the entries' month.
Webhook returns "denied"GAS_SECRET ≠ SHARED_SECRET, or Apps Script STRIPE_SECRET_KEY missing.
fee_pence always 0The fee_pence header isn't in Entries column M, or (rarely) Stripe hadn't finalised the fee at that instant.
Draw runs but picks no winnerCheck Apps Script logs for "CHARITY MINIMUM BREACH" — prize % + expenses too high vs sales. Lower them.
Custom domain stuck "Verifying"Delete any pre-existing draw/lottery CNAME, then let Pages re-create it via Set up a domain.
Code change not taking effectRedeploy Apps Script as Manage deployments → New version (same URL); redeploy the Worker; hard-refresh / purge Cloudflare cache for the page.

Key facts to remember

  • £2 per ticket, same price for everyone, payment before entry, 16+ only.
  • Max £20,000 ticket sales per draw · max prize £25,000 · at least 20% to good causes · no more than 80% on prizes + expenses.
  • Keep records: ≥3 years (lottery) and ~6 years (financial / Gift Aid). Winner bank details deleted once paid.
  • Lottery return to ERYC within 3 months of each draw.
  • Public wording: "50% to the winner, the rest to good causes." Rules page keeps the precise "at least 20%."
